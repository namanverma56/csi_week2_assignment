--Q19. Write an INNER JOIN query to display each order along with the customer's first_name and last_name.
--Show: order_id, order_date, first_name, last_name, total_amount.
SELECT o.order_id, o.order_date, c.first_name, c.last_name, o.total_amount
FROM orders o
INNER JOIN customers c ON o.customer_id = c.customer_id;
--an INNER JOIN allows us to instantly see exactly 
--who placed each order without manually matching IDs between different tables.


--Q20. Using a LEFT JOIN, list ALL customers and their orders (if any).
--Customers with no orders should still appear with NULL values for order columns. 
SELECT c.customer_id, c.first_name, c.last_name, o.order_id, o.status, o.total_amount
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id;


--Q21. Write a query using JOINs across three tables (orders → order_items → products) to show:
--order_id, product_name, quantity, unit_price, and discount_pct for each order item. 
SELECT oi.order_id, p.product_name, oi.quantity, oi.unit_price, oi.discount_pct
FROM order_items oi
INNER JOIN orders o ON oi.order_id = o.order_id
INNER JOIN products p ON oi.product_id = p.product_id;

--Q22. Explain the difference between LEFT JOIN and RIGHT JOIN with an example from this schema. 
--When would you use a FULL OUTER JOIN? 
-- A LEFT JOIN keeps all records from the left table (customers) even if there's no match on the right (orders). A RIGHT JOIN does the exact opposite by preserving the right table. Since SQLite doesn't natively support RIGHT JOIN, we achieve the same result by reversing the order of the tables inside a standard LEFT JOIN statement.  
--FULL OUTER JOIN: You use a FULL OUTER JOIN when you want to see all records from both tables, matching rows where possible, and showing NULL gaps on both sides where there is no overlap. This is useful for combining disjointed lists, like checking web-traffic logs against internal purchase data.
